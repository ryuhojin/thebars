# 변경내역 — 기기별 라우트 제거

## 변경 목적

기존 문서는 화면 크기별로 별도 관리자·고객 경로와 별도 개발 업무를 두었다. 이 방식은 다음 문제를 만들 수 있다.

- 같은 기능의 화면과 상태 로직이 중복됨
- 경로별 기능 차이가 커져 유지보수 비용 증가
- 기기 회전·창 크기 변경 시 URL 이동과 상태 유실 가능
- 프론트엔드와 E2E 테스트가 불필요하게 중복됨
- 권한과 기능 제공 범위를 UI 경로와 혼동할 위험

## 최종 결정

- 관리자 업무는 업무별 URL 하나만 사용한다.
- 고객 메뉴판은 `/{encodedSlug}` 하나만 사용한다.
- 화면 폭은 compact·medium·wide 표현을 선택하는 데만 사용한다.
- 권한, API, 데이터 계약은 화면 폭에 따라 달라지지 않는다.
- 각 업무를 개발할 때 세 화면 폭을 동시에 완료한다.

## 개발계획 변경

- 고객 화면 폭별 별도 업무를 하나의 `D14 고객 메뉴판 반응형 UI`로 통합했다.
- 관리자 화면 폭별 별도 업무를 제거하고 D01~D13, D15~D22의 각 업무 완료 조건에 반응형 구현을 포함했다.
- 주문 화면 폭별 별도 업무를 제거하고 D18~D21에 통합했다.
- 전체 업무를 D00~D24로 재정렬했다.

## 금지되는 구현

- 화면 폭 또는 user-agent에 따른 URL redirect
- 기기 종류를 route segment에 포함
- 같은 업무의 화면 전체 복사본
- `isMobile ? MobilePage : DesktopPage` 형태의 전체 페이지 이중화
- compact 화면은 나중에 처리한다는 완료 주장

## 허용되는 구현

- CSS media query와 container query
- 동일 route 안의 grid/list/stack 전환
- table과 card view가 같은 query·selection·mutation을 공유하는 적응형 표현
- dialog와 full-screen sheet가 같은 form state를 공유하는 표현
- wide split pane과 compact 목록→상세 흐름

## 와이어프레임 통합

- 기존 기기별 화면 복사형 와이어프레임을 폐기했다.
- `wireframes/` 폴더에 단일 URL 반응형 와이어프레임 22개를 추가했다.
- Compact 390, Medium 768, Wide 1440 기준을 하나의 화면 데이터와 컴포넌트에서 전환한다.
- `요구사항.md`, `설계.md`, `개발계획.md`, `AGENTS.md`, `진행상태.md`에 와이어프레임 참조 절차를 반영했다.
- Codex는 각 업무 시작 전 `wireframes/화면맵.md`의 필수 화면을 확인하고 완료 보고에 비교 결과를 남겨야 한다.
