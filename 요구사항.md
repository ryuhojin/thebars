# Bar Menu Operations v1.0 — 최종 요구사항

## 1. 문서 목적과 제품 정의

이 시스템은 실제 소규모 바에서 사용하는 운영 도구다. 다음 세 영역을 하나의 제품으로 제공한다.

1. 바·사용자·카테고리·메뉴를 관리하는 관리자 시스템
2. GitHub와 Cloudflare Pages로 배포되는 정적 고객 메뉴판
3. 별도 POS 없이 계좌이체 정산을 돕는 간이 테이블 주문

예상 규모는 초기 2개 바, 1년 후 약 10개 바이며 바별 정식 메뉴 발행은 평균 주 1회다. 비용을 낮추되 인증, 권한, 금액, 발행 복구, 동시 수정은 데모 수준으로 구현하지 않는다.

## 2. 시스템 범위와 저장소

### 2.1 관리자 시스템

```text
저장소: admin-menu-manager
호스팅: Cloudflare Pages
프론트엔드: React + TypeScript + Vite
API: Cloudflare Pages Functions + Hono
DB: Cloudflare D1
검증: Zod
DB 매핑: Drizzle ORM + 순차 SQL migration
테스트: Vitest + 통합 테스트 + 브라우저 E2E
```

관리자 시스템은 인증, 권한, 바·메뉴, 발행, 테이블 주문·정산, 감사 로그를 담당한다.

### 2.2 고객 메뉴판

```text
저장소: customer-menu-board
호스팅: Cloudflare Pages
데이터: public/menus/{encodedSlug}.json
프론트엔드: React + TypeScript + Vite
```

고객 메뉴판은 정적 JSON만 읽는다. 관리자 API, D1, 주문 API를 호출하지 않는다.

### 2.3 저장소 공개 범위

- 두 GitHub 저장소는 private이다.
- 고객 메뉴판 Pages 사이트만 공개한다.
- 초기 운영은 고객 저장소 한 곳에만 `Contents: write` 권한이 있는 fine-grained PAT를 쓴다.
- PAT는 Cloudflare Secret에만 저장하고 로그·브라우저·DB에 남기지 않는다.
- 업장 확대 전 GitHub App 전환을 운영 개선 항목으로 둔다.

## 3. 사용자, 역할, 권한

### 3.1 시스템 관리자

- 시스템 전체에 1명만 존재한다.
- 최초 `/setup`에서 `SETUP_TOKEN`으로 생성한다.
- 생성 후 `/setup`을 비활성화한다.
- `/recovery`에서 `ADMIN_RECOVERY_TOKEN`으로 비밀번호를 복구한다.
- 바 생성·비활성화, 사용자 계정, 바 소속과 역할, 공통 품목 유형, 공통 배지, 색상, 발행 장애를 관리한다.

### 3.2 바 사용자

역할은 `owner`, `manager`, `staff`다.

- 한 사용자는 여러 바에 소속될 수 있다.
- 역할은 바마다 다를 수 있다.
- 계정 생성, 소속 추가·삭제, 역할 변경은 시스템 관리자만 수행한다.

### 3.3 바별 권한

```text
can_edit_menu
can_manage_orders
can_add_custom_order_item
can_apply_order_adjustment
```

신규 바 기본값:

| 역할 | 메뉴 편집 | 주문 운영 | 기타 주문 항목 | 금액 조정 |
|---|---:|---:|---:|---:|
| owner | 허용 | 허용 | 허용 | 허용 |
| manager | 허용 | 허용 | 허용 | 허용 |
| staff | 불허 | 허용 | 불허 | 불허 |

발행:

- 시스템 관리자는 모든 바를 발행할 수 있다.
- 신규 바의 `direct_publish_enabled`는 false다.
- true인 바에서는 active owner만 발행할 수 있다.
- manager와 staff는 발행할 수 없다.

## 4. 인증과 세션

### 4.1 로그인

- 아이디 + 비밀번호 방식이다.
- 아이디는 전체 시스템에서 유일하고 대소문자를 구분하지 않는다.
- 형식은 `^[a-z0-9]{4,20}$`다.
- 저장·검색용 정규화 아이디는 소문자로 저장한다.

### 4.2 비밀번호

- 최소 10자다.
- 영문, 숫자, 특수문자를 각각 1자 이상 포함한다.
- 시스템 관리자가 일반 사용자 계정과 임시 비밀번호를 생성한다.
- 최초 로그인 또는 비밀번호 초기화 후에는 비밀번호 변경 전까지 다른 기능을 사용할 수 없다.
- 비밀번호는 검증된 비밀번호 해시 알고리즘으로 저장하고 평문을 저장하지 않는다.

### 4.3 로그인 보호와 세션

- 5회 연속 실패 시 15분 잠근다.
- 시스템 관리자가 잠금을 해제할 수 있다.
- 여러 기기 동시 로그인을 허용한다.
- 마지막 인증 활동 후 8시간에 만료되는 sliding session이다.
- 세션 touch는 DB 쓰기 폭증을 막기 위해 최소 5분 간격으로 갱신한다.
- 비밀번호 초기화는 기존 세션을 유지한다.
- 계정 비활성화는 해당 사용자의 기존 세션을 즉시 무효화한다.
- 변경 요청은 CSRF 방어를 적용한다.

## 5. 통합 반응형 UI

### 5.1 단일 URL 원칙

- 관리자와 고객 화면은 기기별 URL, 하위 도메인, 강제 리디렉션을 사용하지 않는다.
- 같은 업무는 화면 폭과 관계없이 같은 URL을 유지한다.
- 화면 크기를 바꾸거나 기기를 회전해도 현재 URL과 작성 중인 상태를 유지한다.
- 사용자 에이전트나 화면 폭으로 별도 사이트를 선택하지 않는다.
- 권한은 화면 크기와 무관하며 서버가 동일하게 검사한다.

관리자 주요 URL 예시:

```text
/setup
/recovery
/login
/change-password
/dashboard
/bars
/bars/new
/bars/{barId}
/bars/{barId}/members
/bars/{barId}/settings
/bars/{barId}/categories
/bars/{barId}/menus
/bars/{barId}/menus/{menuItemId}
/bars/{barId}/preview
/bars/{barId}/publications
/bars/{barId}/orders
/bars/{barId}/orders/{orderTabId}
/bars/{barId}/settlements
/system/users
/system/item-types
/system/badges
/system/audit
```

고객 메뉴판 URL:

```text
/{encodedSlug}
```

### 5.2 화면 폭 구간

구현 토큰의 정확한 값은 설계에서 관리하되 v1 검증 기준은 다음이다.

| 구간 | 기준 폭 | 주 사용 환경 |
|---|---:|---|
| compact | 360~767px | 휴대폰, 작은 창 |
| medium | 768~1199px | 태블릿, 작은 노트북 |
| wide | 1200px 이상 | 데스크톱, 큰 태블릿 가로 |

- 최소 지원 폭은 360px다.
- 페이지 전체에 불필요한 가로 스크롤이 생기면 안 된다.
- 화면 폭만으로 업무 기능을 제거하지 않는다.
- 모든 권한 있는 핵심 작업은 compact에서도 수행 가능해야 한다.
- 넓은 화면의 다중 컬럼·분할 패널은 좁은 화면에서 순차 화면, 탭, 아코디언, drawer 또는 full-screen sheet로 바뀐다.

### 5.3 공통 적응 규칙

- wide의 고정 사이드바는 medium에서 축소형, compact에서 drawer로 바뀐다.
- wide의 데이터 테이블은 compact에서 카드/행 요약 목록으로 바뀌며 핵심 값과 주요 액션을 유지한다.
- 긴 폼은 wide에서 2열, compact에서 1열로 바뀐다.
- modal은 compact에서 full-screen sheet로 표시할 수 있다.
- 주요 저장·발행·정산 액션은 compact에서 하단 sticky action bar로 접근 가능해야 한다.
- drag-and-drop 기능에는 위·아래 이동 버튼 또는 대상 선택 방식이 항상 있어야 한다.
- 최소 touch target은 44×44px다.
- 키보드 focus, label, 오류 요약, 색상 대비를 제공한다.
- 화면 폭 변경 시 저장되지 않은 폼, 선택 항목, 검색·필터 상태를 유실하지 않는다.

### 5.4 기능 제공 범위

기존처럼 기기 종류를 이유로 사용자 관리, 메뉴 생성, 카테고리 편집 등을 별도 경로에서 차단하지 않는다.

- 사용 권한이 있으면 같은 URL에서 기능에 접근할 수 있다.
- compact에서는 복잡한 작업을 단계형 폼 또는 full-screen 편집기로 단순화한다.
- 위험하거나 대량인 작업은 확인 절차를 강화할 수 있으나 단순히 숨기지 않는다.
- 운영 편의를 위해 compact 대시보드에는 품절, 주문, 발행 같은 빠른 액션을 우선 노출한다.

### 5.5 승인 와이어프레임과 UI 기준

- 반응형 UI 기준 파일은 `wireframes/index.html`과 `wireframes/화면맵.md`다.
- 와이어프레임은 관리자 인증·대시보드·바·회원·설정·카탈로그·발행·주문·감사 로그와 고객 메뉴판을 포함한다.
- 화면의 제품 URL, 정보 계층, 핵심 액션, 필수 상태, compact·medium·wide 전환 방식은 구현 요구사항의 일부다.
- 와이어프레임의 회색 계열 색상, 세부 간격, 임시 아이콘은 최종 시각 디자인을 강제하지 않는다.
- 요구사항과 와이어프레임이 충돌하면 이 문서가 우선하며, 주요 정보 구조나 작업 순서를 임의로 바꾸지 않는다.
- 각 프론트엔드 업무는 `wireframes/화면맵.md`에서 연결된 화면을 검토하고 390×844, 768×1024, 1440×900에서 구현 결과를 비교해야 한다.
- 화면 폭 전환으로 URL, 폼 dirty state, 검색·필터, 선택 항목, 미저장 일괄 변경을 잃으면 요구사항 미충족이다.
- 와이어프레임에 표시된 기능이라도 사용자 권한이 없으면 서버가 차단하고 UI는 forbidden 또는 읽기 전용 상태를 명확히 보여야 한다.

## 6. 대시보드

로그인 후 권한에 맞는 대시보드를 제공한다.

시스템 관리자:

- 전체 바 수와 상태
- 최근 발행 실패·확인 불가
- 잠긴/비활성 사용자 알림
- 빠른 바 등록, 사용자 생성 진입

바 사용자:

- 접근 가능한 바 목록 또는 현재 바 선택
- 미발행 변경 여부
- 열린 테이블 수와 계산 요청 수
- 최근 발행 상태
- 권한에 맞는 빠른 작업

반응형 요구:

- wide에서는 요약 카드와 최근 활동을 병렬 배치한다.
- medium에서는 2열 카드로 줄인다.
- compact에서는 우선순위가 높은 카드와 빠른 액션을 1열로 배치한다.
- 같은 `/dashboard` URL을 유지한다.

## 7. 바 관리

### 7.1 생성과 상태

- 바 생성은 시스템 관리자만 가능하다.
- 신규 바는 active 상태다.
- 바 이름, 통화는 필수다.
- 최초 공개 전에는 `preparing` 메뉴판을 제공한다.
- inactive 전환 시 고객 JSON을 삭제하고 고객 URL은 404가 된다.
- D1의 바·메뉴·발행 이력은 유지한다.
- active 복원 시 마지막 성공 공개본을 복원한다. 성공 공개본이 없으면 preparing 상태를 복원한다.

### 7.2 슬러그와 고객 경로

- 생성 시 내부 slug를 자동 생성하고 이후 변경하지 않는다.
- 한글 이름은 `bar-` + 영문 소문자·숫자 6자리 난수를 사용한다.
- 충돌 시 재생성한다.
- 고객 경로와 JSON 파일명은 padding 없는 Base64URL `encodedSlug`를 쓴다.
- encodedSlug는 접근 통제가 아니라 주소 추측 난이도만 높인다.

### 7.3 바 기본 정보

- 이름: 필수
- 소개: 선택, 최대 500자, 줄바꿈 허용
- 주소: 선택 자유 텍스트
- 지도 URL: 선택, http/https만 허용
- 전화번호: 선택, 국내 번호 숫자 저장 및 표시 포맷
- 영업시간: 요일별 복수 구간, 익일 마감 허용
- 영업 안내: 선택
- 자유 링크: 이름 + URL, 최대 5개
- 통화: 생성 시 필수, 시스템 관리자만 변경
- 통화 변경은 기존 금액 숫자를 변환하지 않는다.
- 로고와 이미지는 v1에서 제외한다.

## 8. 사용자와 바별 회원 관리

- 시스템 관리자가 사용자를 생성·조회·비활성화·활성화·잠금 해제·비밀번호 초기화한다.
- 사용자 삭제 대신 비활성화를 기본으로 한다.
- 시스템 관리자가 사용자를 바에 배정하고 owner/manager/staff를 지정한다.
- 바별 역할 권한을 시스템 관리자가 조정한다.
- 다른 바 데이터 접근은 URL 조작을 포함해 서버에서 차단한다.
- compact에서는 회원 테이블을 카드 목록으로 표시하고 편집은 full-screen sheet로 제공한다.

## 9. 카테고리

- 최대 2단계다.
- 이름은 1~30자, 설명은 선택 100자 이하다.
- 설명 공개 여부를 별도 설정하며 기본값은 비공개다.
- 카테고리는 visible/hidden 상태다.
- 숨김 카테고리와 하위 구조는 public JSON에서 제외한다.
- 메뉴가 없는 최종 카테고리도 공개하며 `등록된 메뉴가 없습니다`를 표시한다.
- 하위 카테고리가 있는 상위 카테고리에 메뉴를 직접 넣지 않는다.
- 직접 메뉴가 있는 상위 카테고리에 하위 카테고리를 만들지 못한다.
- 메뉴가 있는 카테고리는 삭제하지 못한다.
- 비어 있는 하위 카테고리만 있는 상위 카테고리는 확인 후 함께 삭제할 수 있다.
- 드래그 앤 드롭과 위·아래 버튼으로 정렬한다.
- 하위 카테고리는 다른 상위로 이동할 수 있으나 구조 규칙을 검증한다.
- compact에서는 트리와 편집 패널을 동시에 강제하지 않고 목록→상세 흐름으로 제공한다.

## 10. 메뉴

### 10.1 기본 정보

- 메뉴 이름: 필수, 1~50자, 같은 바 전체에서 정규화 중복 금지
- 설명: 선택, 최대 200자, 줄바꿈 허용, 공개 시 항상 표시
- 카테고리: 필수
- 품목 유형: 선택
- 판매 상태: `available`, `sold_out`
- 공개 상태: `visible`, `hidden`
- 신규 기본값: available + visible
- ABV: 선택, 0~100, 소수점 둘째 자리까지
- 내부 메모: 공개 JSON 제외, 바 구성원 조회 가능, 시스템 관리자와 owner만 수정

### 10.2 상태와 표시

- hidden 메뉴는 public JSON에서 제외한다.
- sold_out 메뉴는 공개하지만 가격·용량·배지를 숨기고 `품절`을 표시한다.
- sold_out 메뉴의 이름·설명·ABV는 표시한다.
- 고객 화면에서 sold_out을 카테고리 아래쪽으로 정렬하되 관리자 원래 순서는 보존한다.

### 10.3 가격

가격 항목은 0~10개다.

```text
label: 필수, 최대 20자
amountMinor: 필수, 0 이상의 정수
volumeText: 선택, 최대 20자
displayOrder: 명시적 순서
```

- 가격 항목이 없으면 가격 영역을 표시하지 않는다.
- 같은 메뉴 안에서 label 정규화 중복을 금지한다.
- 새 항목은 아래에 추가하고 드래그 또는 이동 버튼으로 정렬한다.
- 단일/다중 가격 모두 동일한 가격 항목 구조로 표현한다.

### 10.4 편집·이동·삭제

- 자동 저장하지 않는다. 저장 버튼으로만 D1에 반영한다.
- 저장하지 않은 상태에서 이동·새로고침·닫기 시 경고한다.
- 목록 빠른 수정과 일괄 수정도 저장 전 클라이언트 변경 상태로 유지한다.
- 일반 메뉴 편집은 마지막 저장 우선 정책을 허용한다.
- 메뉴를 다른 카테고리로 드래그하거나 편집 화면에서 옮길 수 있다.
- 편집 화면 이동은 대상 맨 위, 드래그 이동은 놓은 위치를 사용한다.
- 메뉴 복제와 휴지통은 제공하지 않는다.
- 삭제는 확인 후 영구 삭제하되 과거 발행 스냅샷/Git 기록은 남을 수 있다.
- compact에서는 메뉴 편집 탭을 단계형 섹션으로 표시하고 저장 액션을 고정한다.

## 11. 품목 유형과 정보 템플릿

### 11.1 품목 유형

- 시스템 공통 품목 유형과 바 전용 품목 유형을 제공한다.
- 바 전용 유형은 시스템 관리자와 해당 바 owner가 관리한다.
- 공통 유형은 바별로 숨기거나 가격 label 기본 세트를 재정의할 수 있다.
- 품목 유형 선택은 메뉴에서 선택 사항이다.
- 새 메뉴에서 유형 선택 시 기본 가격 label만 자동 채우며 금액은 입력하지 않는다.
- 유형 변경 시 기존 가격은 유지한다.
- 유형 변경으로 상세 템플릿이 달라지면 확인 후 이전 상세 값을 삭제한다.

### 11.2 고정 템플릿

템플릿 필드와 enum은 코드·계약에 고정하며 관리자 화면에서 구조를 편집하지 않는다.

- 일반
- 와인: 생산자, 국가, 지역/아펠라시옹, 포도 품종, 빈티지, 종류, 당도, 바디, 산도, 탄닌
- 위스키: 브랜드/증류소, 국가, 지역, 분류, 숙성/NAS, 캐스크/피니시, 빈티지/증류연도, 싱글캐스크, 캐스크스트렝스, 논칠필터
- 일반 증류주: 브랜드/생산자, 국가, 지역/원산지, 세부 유형, 주원료, 숙성/등급, 캐스크
- 맥주: 브루어리, 국가, 스타일, IBU, 원재료/향미
- 칵테일: 베이스 주종, 주요 재료, 맛/스타일, 제조 방식, 가니시, 글라스
- 푸드·디저트: 주요 재료, 알레르기, 맵기, 식이 표시, 인분, 페어링
- 시가: 브랜드, 라인, 원산지, 비톨라, 길이, 링 게이지, 래퍼, 바인더, 필러, 강도, 향미, 예상 흡연 시간

모든 전용 필드는 선택이며 입력된 값만 공개한다.

### 11.3 포도 품종

- 공통 승인 목록을 사용한다.
- 메뉴 편집 권한자는 후보를 제출할 수 있다.
- 시스템 관리자가 표준명을 수정해 승인하거나 반려한다.
- 승인 전 후보는 메뉴에 사용할 수 없다.
- 이 정책은 표준화 우선 결정이며 파일럿에서 운영 지연이 확인되면 변경안을 별도 승인한다.

## 12. 배지

- 시스템 공통 배지와 바 전용 배지를 제공한다.
- 메뉴당 최대 3개며 메뉴별 표시 순서를 저장한다.
- 품절 메뉴에서는 배지를 숨긴다.
- 공통 배지는 시스템 관리자만 관리하고 영구 삭제하지 않는다.
- 공통 배지 비활성화 시 모든 메뉴에서 제거하며 재활성화해도 자동 복원하지 않는다.
- 신규 공통 배지는 기존 바에서 기본 숨김이며 신규 바도 모든 공통 배지를 숨김으로 시작한다.
- 바 전용 배지는 시스템 관리자와 메뉴 편집 권한자가 관리한다.
- 사용 중인 바 전용 배지는 영향 건수 확인 후 모든 메뉴에서 제거하고 삭제한다.
- 배지 색상은 시스템 관리자가 관리하며 `#RRGGBB`, `#RRGGBBAA`를 허용한다.
- 글자색은 실제 메뉴판 배경과 합성한 대비를 기준으로 검정/흰색 중 자동 선택한다.

## 13. 관리자 메뉴 목록

두 보기를 제공한다.

- 카테고리 트리 보기
- 전체 메뉴 목록 보기

검색·필터:

- 이름
- 카테고리
- 품목 유형
- 판매 상태
- 공개 상태
- 배지

wide에서는 테이블을 기본으로 사용하고 compact에서는 카드 목록으로 전환한다. 두 표현은 같은 query, selection, mutation 상태를 공유한다.

목록 정보:

- 메뉴명, 카테고리, 품목 유형, 가격, 배지, 판매 상태, 공개 상태, 마지막 수정자, 마지막 수정 시각

빠른 수정:

- 판매 상태, 공개 상태, 정렬

일괄 수정:

- 판매 상태, 공개 상태, 카테고리 이동, 배지 추가·제거

일괄 변경은 영향 건수를 확인한 뒤 화면에 반영하고 최종 저장으로 확정한다. compact에서도 선택 목록과 하단 작업 바를 통해 같은 기능을 제공한다.

## 14. 미리보기와 고객 메뉴판

### 14.1 미리보기

- 관리자 내부에서만 제공한다.
- 저장된 D1 데이터만 사용한다.
- 전체 메뉴판, 특정 카테고리, 특정 메뉴를 미리 본다.
- 고객 렌더러와 public DTO 규칙을 재사용한다.
- 미리보기 화면은 현재 viewport 폭에 맞춰 실제 고객 메뉴판과 같은 반응형 규칙을 적용한다.

### 14.2 고객 단일 라우트

```text
/{encodedSlug}
```

- 별도 모바일·태블릿 URL을 만들지 않는다.
- URL을 바꾸거나 리디렉션하지 않는다.
- 같은 public JSON과 같은 페이지 상태를 유지하면서 화면 폭에 따라 레이아웃만 조정한다.
- hero의 소개 문구는 public JSON `bar.intro`를 표시하며, 비어 있으면 `현재 공개된 메뉴를 확인하세요.`를 표시한다.

compact:

- 단일 컬럼
- 바 정보 접기/펼치기
- 검색
- 가로 스크롤 또는 sticky 카테고리 탐색
- 메뉴 카드와 상세 펼침

medium:

- 상단 바 정보
- 접을 수 있는 카테고리 rail 또는 상단 탐색
- 메뉴 1~2열
- 상세 정보 인라인 또는 sheet

wide:

- 최대 콘텐츠 폭 적용
- 고정 또는 sticky 카테고리 rail
- 메뉴 2열까지 허용
- 상세 정보 패널 또는 인라인 확장

공통:

- loading, preparing, published, 404, schema/error 상태
- 5분 이상 유휴 상태 후 검색·선택 상태를 초기화할 수 있으나 URL은 유지한다.
- 고객 UI에는 주문, 장바구니, 결제, 로그인, 관리자 링크를 제공하지 않는다.

## 15. 발행

### 15.1 파일과 인증

```text
public/menus/{encodedSlug}.json
public/publish-triggers/{encodedSlug}.json
```

- 일반 내용 변경은 메뉴 JSON을 갱신한다.
- 동일 내용 재발행은 trigger 파일을 갱신해 실제 Git 변경과 Pages 배포를 보장한다.
- 고객 저장소 `main`에 고정 메시지로 커밋한다.
- GitHub 쓰기는 저장소 전체에서 직렬화한다.

### 15.2 공개 JSON 메타데이터

```text
schemaVersion
status: preparing | published
revision
publishedAt
contentHash
bar
categories
```

- 내부 DB ID, 내부 메모, 토큰, 사용자 정보는 포함하지 않는다.
- 공개 ID는 바 내부 순번 기반 `cat_1`, `item_1`이며 삭제 후 재사용하지 않는다.
- canonical JSON의 해시로 현재 편집본과 공개본 동일 여부를 판단한다.

### 15.3 발행 흐름과 상태

1. 권한 검사
2. 바별 발행 lock
3. 최신 저장 데이터 조회
4. public DTO와 canonical hash 생성
5. JSON Schema 검증
6. GitHub 전역 commit lock
7. 파일 생성/수정/삭제 commit
8. commit SHA 저장
9. 해당 SHA의 Cloudflare Pages deployment 확인
10. 관리자 클라이언트가 30초마다 상태 조회
11. 최대 3분 확인
12. 성공/실패/확인 불가 기록

상태:

```text
pending
building_json
validating_json
committing_github
waiting_cloudflare
success
failed
timeout_unknown
```

- 자동 재시도하지 않는다.
- `timeout_unknown`은 늦게 실제 배포가 성공할 수 있음을 사용자에게 알린다.
- 발행 중 메뉴 저장은 허용하고 JSON 생성 직전 최신 저장 데이터를 쓴다.

### 15.4 버전과 복구

- 내용 변경 시 revision 증가 및 publishedAt 갱신
- 동일 내용 재발행 시 revision/contentHash/publishedAt 유지
- 성공 publication + snapshot은 바별 최근 100건
- 실패/timeout_unknown은 별도 최근 100건
- 과거 성공 snapshot을 새 커밋으로 재발행할 수 있다.
- 과거 재발행은 현재 D1 편집본을 바꾸지 않는다.
- 공개본과 편집본이 다르면 관리자에서 경고한다.

## 16. 간이 테이블 주문·정산

이 기능은 정식 POS나 회계 시스템이 아니라 계좌이체 업장의 현장 보조 도구다.

### 16.1 테이블 주문

- `/bars/{barId}/orders`는 테이블 목록과 테이블 생성 진입점만 제공한다.
- `/bars/{barId}/orders/new`는 테이블 생성 전용 화면이며 생성 성공 후 목록으로 돌아가 새 테이블을 표시한다.
- `/bars/{barId}/orders/{orderTabId}`는 테이블 상세이며 메뉴 추가, 수량 변경, 계산 요청, 정산 전이를 처리한다.
- `/bars/{barId}/settlements`는 정산 완료된 테이블 정보만 조회한다.
- 테이블 라벨과 간단한 손님 설명으로 테이블 주문 기록을 생성한다.
- 상태는 `open`, `checkout_requested`, `closed`, `cancelled`다.
- closed/cancelled는 수정할 수 없다.
- checkout_requested에 새 주문을 추가하려면 확인 후 open으로 되돌린다.
- 주문 라인이 없거나 모두 void인 테이블만 취소할 수 있다.

### 16.2 주문 라인

종류:

- menu: 메뉴/가격 스냅샷
- custom: 메뉴에 없는 기타 항목
- adjustment: 할인(음수)/추가금(양수)

상태:

- active
- voided

menu snapshot은 메뉴명, 가격 label, 용량, 단가, 통화를 저장한다. 메뉴 가격이 바뀌어도 기존 주문 금액은 바뀌지 않는다.

- 라인을 삭제하지 않고 void 처리한다.
- active 라인만 서버 정수 연산으로 합계를 계산한다.
- 정산 시 최종 합계를 테이블 주문 기록에 고정한다.
- custom/adjustment에는 사유를 기록한다.
- owner/manager는 기본적으로 custom과 adjustment가 가능하고 staff는 기본 불가다.

### 16.3 동시 수정과 중복 명령

- 테이블 주문 기록과 라인은 version 기반 낙관적 잠금을 사용한다.
- stale `expectedVersion`은 409를 반환하고 최신 테이블 주문 기록을 다시 불러오게 한다.
- 주문 추가·정산과 같은 재전송 위험 명령에는 idempotency key를 사용한다.

### 16.4 반응형 주문 UI

같은 주문 URL을 모든 화면 폭에서 사용한다.

- 목록, 생성, 상세는 각각 같은 제품 URL을 모든 화면 폭에서 유지한다.
- wide: 목록은 보드/테이블, 생성은 단일 폼, 상세는 주문 편집 탭과 결제·정산 탭을 분리해 표시한다.
- medium: 목록·생성·상세가 각각 단일 업무 화면으로 전환된다.
- compact: 목록→생성→상세 흐름을 유지하고 상세 안에서 주문 편집과 결제·정산을 탭으로 전환한다.
- 메뉴 검색, 주문 추가, 수량 변경, void, 계산 요청, 정산은 compact에서도 가능해야 한다.
- stale version 충돌은 모든 폭에서 동일한 메시지와 새로고침 동작을 제공한다.

### 16.5 정산과 보관

- 계좌이체 확인 후 closed 처리한다.
- 카드 결제, PG, 영수증, 세금계산서, 재고, 고객 QR 주문은 제외한다.
- open/checkout_requested는 종료까지 보관한다.
- closed/cancelled 테이블·라인·이벤트는 1년 보관한다.
- 일별 정산 요약은 3년 보관한다.
- v1 정리 작업은 시스템 관리자용 maintenance 명령으로 실행 가능하게 하고, 자동 스케줄은 별도 운영 개선으로 둔다.

## 17. 공통 규칙

### 17.1 이름 정규화

중복 제한 이름은 다음 순서로 비교한다.

1. Unicode NFKC
2. 앞뒤 공백 제거
3. 연속 내부 공백 한 칸
4. 영문 소문자화

검사 범위는 엔티티별로 다르다.

### 17.2 공개 텍스트와 URL

- HTML/Markdown 입력을 지원하지 않는다.
- 출력은 항상 escape한다.
- 외부 URL은 http/https만 허용한다.

### 17.3 감사 로그

다음 중요 작업은 작업자, 바, 시각, 요청 ID, 대상, 결과를 기록한다.

- 로그인 실패·잠금 해제
- 계정/회원/권한 변경
- 바 활성/비활성
- 발행·재발행
- 주문 void, adjustment, 정산, 취소
- 중요 마스터 데이터 변경

민감정보와 전체 메뉴 설명·내부 메모·토큰은 로그에 남기지 않는다. 화면 폭이나 기기 종류는 권한 근거로 저장하거나 사용하지 않는다.

## 18. 비기능 요구사항

### 18.1 보안

- 모든 bar-scoped API는 bar ID와 active membership/permission을 함께 검사한다.
- mutation은 CSRF를 검사한다.
- 세션 쿠키는 HttpOnly, Secure, 적절한 SameSite를 적용한다.
- 비밀번호·토큰·세션 원문을 로그에 남기지 않는다.
- 계정 비활성화 시 세션을 즉시 무효화한다.

### 18.2 정확성

- 금액은 정수 최소 통화 단위로 저장한다.
- 주문 합계는 서버에서 계산한다.
- public JSON은 schema 검증 후에만 커밋한다.
- 주문 동시 수정은 409로 충돌을 노출한다.

### 18.3 가용성

- 관리자 장애가 기존 고객 메뉴판을 중단시키지 않아야 한다.
- 발행 실패는 마지막 성공 고객 메뉴판을 최대한 보존한다.
- 외부 API 오류를 사용자에게 단계별로 보여준다.

### 18.4 반응형·접근성

- 대표 검증 viewport는 390×844, 768×1024, 1440×900이다.
- 각 업무의 E2E 핵심 흐름을 세 viewport에서 확인한다.
- 브라우저 resize 후 URL과 저장 전 상태가 유지되어야 한다.
- 키보드 조작, 명확한 focus, touch target, 오류 요약을 제공한다.
- 데이터 밀도는 화면 폭에 따라 달라질 수 있으나 기능과 권한은 달라지지 않는다.
- destructive action은 영향과 결과를 확인시킨다.

### 18.5 성능

- 초기 화면은 현재 업무에 필요한 데이터만 요청한다.
- 긴 목록은 서버 페이지네이션 또는 점진적 로딩을 사용한다.
- 화면 폭별로 동일 데이터를 중복 요청하지 않는다.
- 고객 메뉴판은 정적 자산과 단일 JSON 요청을 중심으로 유지한다.
