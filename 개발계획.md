# Bar Menu Operations v1.0 — 체크리스트 개발계획

## 1. 실행 원칙

- 한 번에 업무 ID 하나만 구현한다.
- 각 업무는 **계약/DB → 백엔드 → 백엔드 테스트 → 프론트엔드 → 반응형 검증 → E2E/인수 → 전체 검증** 순서로 완료한다.
- 모든 프론트 업무는 같은 URL에서 compact·medium·wide를 함께 완료한다.
- 기기 전용 route, 화면 복사본, 후속 반응형 전담 업무를 만들지 않는다.
- 하나의 업무가 완료되기 전 다음 업무를 시작하지 않는다.

## 2. 공통 완료 Gate

각 업무는 적용 가능한 다음 항목을 모두 통과해야 `[x]`다.

### 계약·DB·백엔드

- [ ] 업무 규칙과 요청/응답 contract 확정
- [ ] 필요한 migration/schema 작성 및 빈 DB 적용 검증
- [ ] repository와 application service 구현
- [ ] API route/middleware 구현
- [ ] 정상, 입력 오류, 미인증, 권한 없음, tenant 격리, 충돌 테스트

### 프론트엔드

- [ ] `wireframes/화면맵.md`에서 현재 업무의 필수·참고 화면 확인
- [ ] `wireframes/index.html`에서 Compact·Medium·Wide 정보 구조 검토
- [ ] 업무 route 하나 구현
- [ ] loading, empty, error, forbidden, conflict 상태 구현
- [ ] dirty state와 이탈 경고 구현(해당 시)
- [ ] compact 390×844 동작
- [ ] medium 768×1024 동작

- [ ] 같은 URL 유지 및 resize 상태 보존

- [ ] 페이지 전체 불필요한 가로 스크롤 없음
- [ ] 와이어프레임 대비 주요 정보 구조·액션·상태 일치 확인
- [ ] 의도적으로 달리 구현한 항목과 이유 기록

### 최종 검증

## 2.1 와이어프레임 참조 Gate

프론트엔드가 포함된 모든 업무는 `wireframes/화면맵.md`의 업무별 필수 화면을 구현 기준으로 사용한다.

- [ ] 현재 업무의 와이어프레임 ID와 제품 URL을 완료 보고에 기록
- [ ] `index.html#screen={화면키}&viewport=compact` 확인
- [ ] `index.html#screen={화면키}&viewport=medium` 확인
- [ ] `index.html#screen={화면키}&viewport=wide` 확인
- [ ] 구현 screenshot 또는 E2E 결과를 390·768·1440 폭으로 비교
- [ ] 와이어프레임의 loading/empty/error/forbidden/conflict 요구를 구현
- [ ] 차이가 있으면 요구사항·설계 근거와 함께 `진행상태.md`에 기록

화면 색상과 장식은 저충실도 예시이므로 그대로 복사할 필요가 없다. 제품 URL, 정보 계층, 주요 작업 흐름, 상태와 반응형 전환은 임의로 축소하지 않는다.

### 업무별 화면 매핑

상세 표는 `wireframes/화면맵.md`가 source of truth다. 주요 연결은 다음과 같다.
