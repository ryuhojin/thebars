# Bar Menu Operations v1.0 — 시스템 설계

## 1. 설계 원칙

1. **수직 슬라이스**: 기능은 계약·DB·백엔드·프론트엔드·반응형·테스트를 한 업무로 완결한다.
2. **단일 라우트 반응형**: 같은 업무는 화면 폭과 관계없이 같은 URL과 상태를 쓴다.
3. **정적 고객 경로**: 고객 메뉴판은 관리자 장애와 분리한다.
4. **서버 권한 우선**: UI에서 숨기는 것은 보안이 아니며 API가 항상 재검사한다.
5. **명시적 상태 전이**: 발행과 주문은 상태 머신으로 제한한다.
6. **금액·공개본 스냅샷**: 나중의 메뉴 수정이 과거 주문·발행 이력을 변경하지 않는다.
7. **실패 가시성**: 외부 연동 실패를 덮지 않고 단계·요청 ID·재시도 방법을 보여준다.
8. **작은 운영 규모에 맞춘 단순성**: 2~10개 바를 기준으로 하되 데이터 격리와 복구는 생략하지 않는다.

## 4. 모듈 경계

### Identity & Access

사용자, 비밀번호, 잠금, 세션, setup/recovery, membership, role permission을 담당한다.

### Bar Administration

바 상태, slug, encodedSlug, 통화, 주소, 영업시간, 링크를 담당한다.

### Catalog

카테고리, 메뉴, 가격, 품목 유형, 템플릿, 품종, 배지를 담당한다.

### Publication

public DTO, canonical hash, schema 검증, GitHub commit, Cloudflare deployment, snapshot, 재발행을 담당한다.

### Customer Menu

public JSON parser, 검색, 반응형 렌더러와 오류 상태를 담당한다.

### Order Tabs

테이블/손님 탭, 메뉴 스냅샷, custom/adjustment, 합계, 정산, 낙관적 잠금을 담당한다.

### Operations

감사 로그, request ID, 유지보수, 데이터 보관, 장애 관측을 담당한다.

모듈 간 업무 규칙을 repository나 route handler에서 우회하지 않는다. 상태 전이는 application service에서 수행한다.

## 5. 관리자 요청 처리 파이프라인

```text
request
→ request ID
→ security headers
→ session authentication
→ forced password-change gate
→ CSRF (mutation)
→ tenant/membership/permission guard
→ Zod input validation
→ application service
→ repository/integration adapter
→ standardized response
→ audit/event record
```

### 5.1 응답 규격

성공:

```json
{
  "data": {},
  "meta": { "requestId": "..." }
}
```

오류:

```json
{
  "error": {
    "code": "BAR_NOT_FOUND",
    "message": "바를 찾을 수 없습니다.",
    "fieldErrors": {}
  },
  "meta": { "requestId": "..." }
}
```

주요 HTTP 상태:

- 400 입력 오류
- 401 인증 필요
- 403 권한 없음
- 404 tenant 범위 내 대상 없음
- 409 중복/상태 전이/낙관적 잠금 충돌
- 422 업무 규칙 위반
- 429 로그인·민감 API 제한
- 502 외부 서비스 실패

## 6. 인증·세션 설계

### 6.1 비밀번호와 로그인

- 사용자 테이블에 정규화 아이디와 비밀번호 hash를 저장한다.
- 로그인 실패 수, 잠금 해제 시각을 서버에서 갱신한다.
- 성공 시 실패 수를 초기화한다.
- 평문 임시 비밀번호는 생성 응답에서 한 번만 보여주고 저장하지 않는다.

### 6.2 세션

- 난수 세션 토큰은 쿠키에, DB에는 토큰 hash만 저장한다.
- 세션은 사용자, 생성 시각, 마지막 touch, 만료, 무효화 시각을 가진다.
- 인증 요청마다 만료를 확인한다.
- 마지막 touch 후 5분 이상일 때만 sliding expiry를 DB에 기록한다.
- 계정 비활성화 시 사용자의 active session을 일괄 무효화한다.

### 6.3 Setup/Recovery

- 시스템 관리자 수가 0명일 때만 setup을 허용한다.
- setup/recovery token은 Secret에서 상수 시간 비교한다.
- 민감 endpoint에는 rate limit과 감사 로그를 둔다.

## 7. 권한 설계

모든 bar-scoped service 입력에는 `actorUserId`, `barId`를 명시한다.

```text
system admin
OR
active membership + required permission
```

발행은 일반 permission 컬럼과 별도로 다음 규칙을 쓴다.

```text
system admin
OR
bar.direct_publish_enabled = true
AND membership.role = owner
AND membership active
```

주문 custom/adjustment는 각각 별도 permission으로 검사한다.

## 8. 데이터 모델

모든 시간은 UTC ISO 표현으로 API에 노출하고 DB는 일관된 정수 epoch 또는 ISO 형식 중 하나로 통일한다. 금액은 정수 최소 통화 단위다.

## 9. 트랜잭션과 동시성

- 외부 GitHub/Cloudflare 호출 중 D1 transaction을 열어 두지 않는다.
- DB 상태 확보 → 외부 호출 → 결과 반영 단계로 분리한다.
- 바별 발행 lock과 저장소 commit lock을 구분한다.
- 주문은 `expectedVersion` 조건부 update로 409 충돌을 만든다.
- idempotency key는 actor, bar, operation scope와 함께 유일하게 저장한다.

## 10. 단일 라우트 반응형 프론트엔드 설계

### 10.1 핵심 원칙

- 업무 route component는 하나다.
- 화면 폭별로 route를 복제하거나 리디렉션하지 않는다.
- 동일 query cache, form state, selection state, mutation을 공유한다.
- CSS media query와 container query를 우선한다.
- JavaScript media query는 dialog/sheet, virtualized list 등 상호작용 방식 선택에만 제한적으로 사용한다.
- 렌더링 트리를 완전히 이중화하는 `isMobile ? MobilePage : DesktopPage` 패턴은 금지한다.
- 화면 resize 시 컴포넌트 remount로 dirty form을 잃지 않게 한다.

### 10.2 반응형 토큰

```text
compact: max-width 767px
medium: 768px ~ 1199px
wide: min-width 1200px
content max width: 1440px
minimum supported width: 360px
minimum touch target: 44px
```

토큰은 CSS custom property와 공통 utility에서 관리한다. feature가 임의 breakpoint를 추가하지 않는다.

### 10.3 적응형 컴포넌트

| 컴포넌트 | wide | medium | compact |
|---|---|---|---|
| `AppShell` | 고정 sidebar + header | 축소 sidebar | drawer + compact header |
| `DataView` | table | table 또는 compact table | card list |
| `MasterDetail` | 분할 pane | 전환 pane | 목록→상세 navigation |
| `FormLayout` | 2열 section | 1~2열 | 1열 |
| `Dialog` | centered modal | modal | full-screen sheet |
| `ActionBar` | 상단/인라인 | 상단 | 하단 sticky |
| `CategoryNavigator` | tree/rail | collapsible rail | horizontal chips 또는 목록 |

적응형 표현 사이에서 기능 누락이 없어야 한다. 카드 보기에도 테이블의 핵심 필드, 권한 있는 액션, 선택 상태가 있어야 한다.

### 10.5 상태 보존

- query string에는 업무상 공유 가능한 검색·필터·페이지를 저장할 수 있다.
- viewport 정보는 query string에 넣지 않는다.
- resize 중 router navigation을 발생시키지 않는다.
- 폼 dirty state, 선택된 행, 미저장 일괄 변경은 동일 route에서 유지한다.
- 좁은 화면에서 상세를 열 때 back 동작은 URL 또는 명시적 local navigation 정책 중 업무별로 일관되게 정한다.

### 10.6 와이어프레임 기반 구현 절차

`wireframes/index.html`은 단일 URL 반응형 정보 구조의 승인 기준이며 `wireframes/화면맵.md`가 개발 업무와 화면을 연결한다.

프론트엔드 업무는 다음 순서를 따른다.

1. 현재 업무 ID에 연결된 필수·참고 와이어프레임을 확인한다.
2. 화면의 사용자 목표, 핵심 액션, 필수 상태와 제품 URL을 구현 계약에 반영한다.
3. 공통 feature state와 API query/mutation을 먼저 정의한다.
4. Wide 표현을 그대로 축소하지 않고 동일 상태를 사용하는 adaptive component로 Medium·Compact 표현을 구성한다.
5. 390×844, 768×1024, 1440×900 screenshot 또는 E2E 증거를 남긴다.
6. 와이어프레임과 달라진 점이 있으면 완료 보고와 `진행상태.md`에 이유를 기록한다.

와이어프레임은 저충실도이므로 색상과 장식은 디자인 토큰으로 개선할 수 있다. 다만 정보 계층, 위험 작업의 확인, 저장·발행·정산 액션 위치, table→card와 dialog→sheet 같은 전환 원칙은 임의로 생략하지 않는다. 요구사항과 충돌하면 요구사항을 따르고 충돌 내용을 문서화한다.

## 12. 고객 메뉴판 설계

### 12.1 데이터 로딩

```text
GET /menus/{encodedSlug}.json
→ schemaVersion 확인
→ parser/validator
→ preparing 또는 published renderer
```

고객 UI는 관리자 API를 호출하지 않는다.

### 12.2 단일 URL 렌더링

고객 URL은 `/{encodedSlug}` 하나다.

- compact: 상단 바 정보, sticky category chips, 1열 메뉴 카드
- medium: 접을 수 있는 category rail, 1~2열 메뉴 카드
- wide: sticky category rail, 최대 2열 메뉴, 넓은 상세 영역
- 동일 검색어, 카테고리 선택, 펼침 상태를 resize 중 유지한다.
- 5분 idle reset은 UI 상태만 초기화하고 URL을 바꾸지 않는다.

### 12.3 고객 상태

- loading
- preparing
- published
- not found
- incompatible schema
- malformed data/network error

오류가 있어도 관리자 API나 내부 경로 링크를 노출하지 않는다.

## 13. Public DTO와 canonical hash

공개 DTO 생성 순서:

1. 바 공개 정보 읽기
2. visible 카테고리와 하위 구조 정렬
3. visible 메뉴만 포함
4. sold_out 공개 정렬 적용
5. 가격·배지·템플릿 공개 규칙 적용
6. public ID 사용
7. canonical serializer
8. content hash
9. JSON Schema 검증

canonical hash 대상에서 발행 시도 메타데이터는 제외한다. 동일 콘텐츠 재발행은 trigger 파일을 갱신한다.

## 14. 발행 오케스트레이션

```text
POST publish
→ publication row pending
→ bar lock 획득
→ 최신 D1 snapshot/DTO 생성
→ schema 검증
→ repository commit lock 획득
→ GitHub file SHA 조회
→ menu 또는 trigger commit
→ locks 해제
→ commit SHA 기준 deployment 조회
→ client polling
→ success / failed / timeout_unknown
```

- lock에는 owner token과 lease expiry를 둔다.
- stale lock 회수 규칙을 둔다.
- commit SHA와 deployment source SHA를 비교한다.
- terminal publication을 재사용하지 않는다.

## 15. 주문 상태 머신

```text
open
 ├─ checkout_requested
 │    ├─ open (추가 주문 확인)
 │    └─ closed (정산)
 ├─ closed
 └─ cancelled (유효 주문 없음)
```

주문 라인:

```text
active → voided
```

- closed/cancelled 탭은 immutable이다.
- 정산 총액은 active 라인 합계를 다시 계산해 고정한다.
- custom/adjustment 권한과 사유를 검사한다.

## 17. 캐시와 정적 배포

- 고객 HTML/JS/CSS는 fingerprinted asset으로 장기 캐시한다.
- `/menus/*.json`은 재검증 중심 cache header를 사용한다.
- 고객 앱은 JSON fetch 실패와 schema mismatch를 구분한다.
- 발행 성공 판단은 단순 최신 deployment가 아니라 대상 commit SHA와 연결한다.

## 18. 로깅과 감사


화면 폭이나 기기 구분은 권한·업무 상태에 사용하지 않는다. 필요하면 성능 관측을 위한 비식별 client hint를 별도 분석에 쓸 수 있으나 감사 로그의 핵심 계약에는 포함하지 않는다.

## 21. 주요 위험과 완화

| 위험 | 완화 |
|---|---|
| 반응형 페이지가 기기별 중복 코드로 갈라짐 | 단일 route, shared feature state, adaptive primitive 강제 |
| resize 때 폼 상태 유실 | route/component remount 금지, state ownership 테스트 |
| compact에서 기능 누락 | 각 업무 DoD에 3 viewport 인수 조건 포함 |
| 품절 반영이 즉시 아님 | v1은 발행 필요를 UI에 명시 |
| GitHub 쓰기 충돌 | 전역 commit lock |
| 지연 deployment | timeout_unknown과 SHA 추적 |
| 주문 동시 수정 | version 409 |
| 주문 재전송 | idempotency key |
| PAT 유출 | 최소 권한 Secret, 로그 금지, App 전환 계획 |
| D1 누적 | 보관 정책과 maintenance |
